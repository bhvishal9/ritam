## Observations

- **Factual works.** 0.889 across 45 rows is a healthy starting point. The five factual misses are all in the `f_med_*` and `f_hard_*` buckets — the easy bucket is at 1.000, as it should be.
- **Multi-hop is structurally broken at this top-k.** Several failed multi-hop rows return three chunks from the same doc (e.g. `m_polymer_industry_and_compact`, `m_elastos_processing` — three chunks all from `isoprene_planetary_survey.md`). This means `top_k=3` is retrieving chunks without per-doc deduplication, so a long doc with multiple hot chunks occupies every slot. Until the retriever either dedupes by doc or applies a max-per-doc cap, multi-hop recall is bounded well below 1.0.
- **Out-of-scope abstention is impossible at this threshold.** With 0.70, no query trips abstention; the retriever returns 3 docs for every query, including all 15 unanswerable ones. The bucket is unmeasurable in this configuration.
- **Old short docs are competing with new long ones.** Several factual misses returned `the_duck_wars.md` (15 lines, mostly duplicated by `duck_wars_extended_history.md`) instead of the doc that actually contains the answer. The small originals are now dense noise stealing top-k slots from the docs with the real content.

## Conclusion

0.70 is the right operating point if recall is the only objective and abstention doesn't matter. It is *not* the right operating point for a production-shaped RAG service that needs to refuse to answer questions it cannot ground. The trade-off is explored in the companion runs at 0.80 and 0.85.

This run is the current working baseline for the expanded 78-query dataset.
